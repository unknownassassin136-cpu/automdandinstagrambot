# Disaster Recovery Runbook

## Overview
This document outlines the standard operating procedures for recovering the SaaS platform from critical failures, data loss, or server corruption.

## 1. Database Restoration
We use `pg_dump` to perform regular backups of the PostgreSQL database.

### 1.1 Triggering a Manual Backup
To manually backup the current database state, run the following command from the root directory:
```bash
npm run backup
# OR
pnpm run backup
```
The script will output a file located in `apps/backend/backups/backup-{timestamp}.sql`.

### 1.2 Restoring from a Backup
In the event of a catastrophic database failure, data can be restored using `pg_restore`.

1. **Ensure the database is running but empty** (or clear the corrupted tables).
2. Run the following command:
```bash
pg_restore -d "postgres://postgres:postgres@localhost:5432/instagram_saas" -c backups/backup-{timestamp}.sql
```
*(Replace the connection string with your actual Supabase URL if restoring to production).*

## 2. Server Crash Recovery
We use **PM2** (or Docker) to manage our backend processes in production.

If the Express server crashes unexpectedly:
1. SSH into the production server.
2. Check the PM2 logs:
```bash
pm2 logs backend
```
3. Look for the error generated by `ErrorTrackingService`.
4. Restart the process:
```bash
pm2 restart backend
```

## 3. Redis / BullMQ Failure
If the background queue (BullMQ) fails or Redis goes down:
1. Instagram Webhooks will begin returning `500` errors, which triggers Meta's exponential backoff retry system.
2. Restart the Redis service immediately.
3. The queue will automatically reconnect to Redis once it is available.
4. Any unprocessed webhooks during the downtime will be retried by Meta automatically for up to 7 days.
